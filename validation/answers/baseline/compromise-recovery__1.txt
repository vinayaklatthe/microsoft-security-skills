Hardening after eviction is about closing the doors the attacker used and the persistence paths they could re-use. Assume they still know your usernames, your domain, and roughly how you operate. Here is what I would do.

1. Lock down privileged identity
- Move all privileged roles to just-in-time activation with Entra ID PIM. No standing Global Admin.
- Require approval and phishing-resistant MFA for activation of high-impact roles.
- Keep the number of Global Admins very small (aim for under 5) and use least-privilege roles instead where possible.
- Maintain 2 break-glass accounts, excluded from Conditional Access, with FIDO2 keys, stored securely, and alert on any sign-in.
- Use separate cloud-only admin accounts (not synced from on-prem, not used for email/browsing).

2. Enforce strong authentication everywhere
- Require MFA for all users via Conditional Access; require phishing-resistant methods (FIDO2, Windows Hello for Business, certificate) for admins.
- Block legacy authentication protocols (they bypass MFA).
- Turn on security defaults only if you are not using Conditional Access; otherwise build proper CA policies.
- Review and tighten authentication methods policy; remove SMS/voice for admins.

3. Clean and control applications and consent
- Review every enterprise app and app registration. Remove unused ones.
- Turn off user consent or restrict it to verified publishers and low-risk permissions; require admin consent workflow.
- Audit all app credentials (secrets/certs), remove unknown ones, and set expiry.
- Watch for new credentials added to service principals (a classic persistence trick).

4. Verify the trust fabric
- Re-check federated domains - remove any you do not recognise. Federation backdoors let attackers forge tokens.
- Review Conditional Access policies and named locations for attacker-added allowlists.
- Review delegated/partner (GDAP) relationships.
- If hybrid, secure Entra Connect: harden the sync server as a Tier 0 asset, rotate the sync account, and review password hash sync / pass-through auth / seamless SSO settings.

5. Turn on detection and monitoring
- Enable Entra ID Protection (risky users/sign-ins) and act on risk with Conditional Access.
- Send Entra ID, audit, and sign-in logs to a SIEM (Sentinel) with adequate retention.
- Enable Defender for Cloud Apps / Defender XDR and alert on privileged role changes, new federated domains, app credential additions, and mass mailbox rule creation.
- Alert on break-glass account usage.

6. Reduce the blast radius
- Apply least privilege across Azure RBAC and Entra roles.
- Separate identity tiers (Tier 0 admin workstations / PAWs for admin tasks).
- Review guest access and external collaboration settings.
- Rotate any secrets, certs, and service-account credentials that could have been exposed.

7. Validate and rehearse
- Run an access review of all privileged roles and guests.
- Confirm Secure Score improvements and re-test that break-glass works.
- Document the incident and run a tabletop so the team can respond faster next time.

The themes are: no standing privilege, phishing-resistant MFA, no legacy auth, tight app-consent control, a clean trust fabric (federation/sync), and real monitoring on the actions an attacker would need to come back.
