To require phishing-resistant MFA for all Global Admins, you combine an Authentication Strength with a Conditional Access policy scoped to the privileged roles. Here is how.

Background
- "Phishing-resistant MFA" means methods that cannot be defeated by phishing/AITM proxies: FIDO2 security keys, Windows Hello for Business, and certificate-based authentication (smart cards). SMS, voice, and push approvals are not phishing-resistant.
- Entra ID ships a built-in Authentication Strength called "Phishing-resistant MFA" that includes exactly these methods.

Steps

1. Make sure admins can register a phishing-resistant method
- Ensure the authentication methods policy allows FIDO2 security keys (and/or Windows Hello / certificate-based auth).
- Have each Global Admin register a method (e.g. enrol a FIDO2 key) before you enforce, or they will be locked out.

2. Create the Conditional Access policy
- Entra ID > Protection > Conditional Access > New policy.
- Name: e.g. "CA - Require phishing-resistant MFA for admins".
- Assignments > Users: select Directory roles, and include Global Administrator (and ideally the other high-impact roles: Privileged Role Admin, Security Admin, Exchange/SharePoint/Application Admin, etc.).
- Exclude your break-glass / emergency access accounts (or ensure they also have FIDO2 keys; at minimum verify they will not be locked out).
- Target resources: All cloud apps (or All resources).
- Grant: Require authentication strength > Phishing-resistant MFA.

3. Roll out safely
- Set the policy to Report-only first and review the sign-in logs to confirm which admins would be blocked (i.e. have not yet registered a phishing-resistant method).
- Once all admins are registered and impact looks clean, switch the policy On.

4. Validate
- Test with a non-break-glass admin account: confirm sign-in requires the FIDO2 key / Windows Hello and that password+SMS/push is refused.
- Confirm break-glass accounts still work.

5. Strengthen further (optional but recommended)
- Pair with PIM so admin roles are just-in-time and require this authentication strength on activation.
- Require compliant/managed devices for admin sign-in.
- Consider scoping the policy to "All resources" including the Microsoft Admin Portals to cover management surfaces.

Key cautions: register the phishing-resistant method first, exclude/secure break-glass accounts, and use report-only before enforcing. Targeting the policy at the Global Administrator directory role (not a static user list) ensures anyone who becomes a Global Admin is automatically covered.
