Isolating vaults and keeping secrets fresh are two sides of good Key Vault hygiene. Here is how I approach both.

Isolating vaults
- One vault per application per environment. Do not share a single vault across unrelated apps or across dev/test/prod. This limits blast radius if one identity or environment is compromised.
- Separate vaults by trust boundary and by data sensitivity. Keep high-value keys (e.g. encryption keys, signing keys) in their own vault, separate from application secrets.
- Use separate vaults (or Managed HSM) for keys that need FIPS 140-2 Level 3 / HSM protection.
- Scope access tightly with RBAC. Assign data-plane roles (Key Vault Secrets User, Crypto User, etc.) at the individual vault scope, never at subscription or resource-group scope.
- Network isolation: disable public network access and use private endpoints, or restrict to specific VNets/IPs with the firewall. Keep prod vaults off the public internet.
- Use managed identities for access so there is no bootstrap secret to protect.
- Separate management plane from data plane. Few people should have Contributor/owner on the vault; most identities only need data-plane read.

Keeping secrets fresh (rotation)
- Prefer to eliminate secrets entirely: use managed identities and Entra ID authentication instead of stored connection strings/passwords wherever the service supports it.
- For secrets you must keep, automate rotation:
  - Many Azure resources support Key Vault-integrated rotation; some database/storage secrets can be rotated with an Event Grid + Function pattern.
  - Set an expiry date on every secret and key (--expires) so a missed rotation is visible.
  - Use Event Grid "near expiry" events to trigger an Azure Function that rotates the credential at the source and writes the new value back to the vault.
- Version secrets rather than overwriting blindly. Key Vault keeps versions; apps should read the current version and tolerate a brief overlap during rotation (dual-credential / two-secret strategy) so there is no downtime.
- Have apps refresh cached secrets periodically (e.g. on a timer or on auth failure) rather than reading once at startup, so rotation takes effect without redeploying.
- Rotate keys used for encryption on a schedule and enable auto-rotation for supported key types.

Operational controls
- Enable soft-delete and purge protection so a deleted/rotated secret can be recovered.
- Turn on diagnostic logging to Log Analytics and alert on unusual secret reads, failed access, and secrets nearing expiry.
- Run periodic access reviews on who and what can read each vault.
- Track secret expiry centrally and alert before anything expires to avoid outages.

In short: many small, tightly-scoped vaults instead of one big shared one; managed identity over stored secrets; expiry dates plus automated, overlap-friendly rotation; and logging/alerting so nothing silently goes stale.
