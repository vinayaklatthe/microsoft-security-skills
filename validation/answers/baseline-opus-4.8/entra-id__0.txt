Designing the identity foundation for a new Entra ID tenant means getting the structural and security decisions right from day one, because they are hard to change later. Here is how I would lay it out.1. Tenant and naming basics- Decide tenant type (workforce vs external/B2C) and configure the verified custom domain(s).- Set the tenant's default region/data location and branding.- Plan environment separation (separate tenants for prod vs non-prod where justified).2. Privileged access model (do this early)- Create 2 break-glass / emergency access cloud-only accounts, with strong unique passwords and FIDO2 keys, excluded from Conditional Access, stored securely and monitored.- Use Entra ID PIM for all admin roles: assignments are eligible (just-in-time), not standing.- Keep Global Admins to a small number (under 5) and use least-privilege roles for everything else.- Use dedicated cloud-only admin accounts separate from day-to-day user accounts.3. Authentication and MFA- Require MFA for all users; require phishing-resistant MFA (FIDO2, Windows Hello for Business, certificate) for admins.- Configure the authentication methods policy; prefer passwordless. Disable SMS/voice for admins.- Block legacy authentication protocols.- Enable self-service password reset (SSPR) with strong verification, and combined registration.4. Conditional Access baseline- Build a baseline set of policies: require MFA for all users, require MFA/phishing-resistant for admins, block legacy auth, require compliant or hybrid-joined devices for sensitive access, and risk-based policies if you have P2.- Use report-only mode first, exclude break-glass accounts, and roll out in rings.5. Identity Protection (P2)- Enable risky user and risky sign-in detection and tie remediation to Conditional Access (e.g. require password change / MFA on risk).6. Device identity strategy- Decide between Entra-joined (cloud-first, recommended for new), hybrid-joined, or registered devices.- Integrate with Intune for device compliance to feed Conditional Access.7. User and group strategy- Plan groups: security groups, Microsoft 365 groups, dynamic groups, and role-assignable groups for admin delegation.- Use group-based licensing and access assignment.- Define a joiner/mover/leaver process.8. Application and consent governance- Restrict user consent to low-risk permissions / verified publishers, and enable the admin consent workflow.- Plan how apps authenticate (managed identities, app registrations with certificates over secrets).9. External collaboration / guest access- Configure B2B external collaboration settings and cross-tenant access policies.- Decide guest invitation permissions and apply access reviews to guests.10. Governance and lifecycle (Entra ID Governance, P2)- Set up access reviews for privileged roles and guests.- Consider entitlement management (access packages) for scalable access requests.- Plan access certification cadence.11. Monitoring and logging- Send sign-in and audit logs to Log Analytics / Sentinel with appropriate retention.- Alert on privileged role changes, risky sign-ins, new federated domains, and break-glass usage.- Track Microsoft Secure Score and work the recommendations.12. Hybrid (if applicable)- If syncing from on-prem AD, plan Entra Connect / Cloud Sync, choose an auth method (password hash sync recommended for resilience), and harden the sync server as Tier 0.Sequence it: structure and domains first, then privileged access (PIM + break-glass), then strong auth and Conditional Access, then devices, governance, and monitoring. Getting privileged access and MFA right from the start is what prevents the common day-one mistakes.
